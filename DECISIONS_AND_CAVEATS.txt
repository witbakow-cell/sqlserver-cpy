sqlserver-cpy - Decisions, Observations, and Caveats
=====================================================

This file records architectural choices and practical caveats for the project.
It is intentionally plain text for easy browsing and diffing.

Decisions
---------

1. Dependency strategy: dbatools-first.
   - Use dbatools cmdlets as the primary engine for SQL Server automation tasks.
   - Fall back to the SqlServer PowerShell module where dbatools coverage is
     incomplete (e.g. parts of SSISDB object management).
   - Fall back further to SMO / AMO / TOM / ADOMD.NET directly only when neither
     module exposes the needed surface.

2. UI style: interactive console TUI.
   - Menu-driven flow started from Start-SqlServerCopy.ps1.
   - Each action logs progress to the screen.
   - On uncaught runtime errors the run ends with an error screen rather than a
     stack trace dump; the user is prompted whether to copy the full error.
     Clipboard copy uses Set-Clipboard when available; otherwise the full error
     is written to a timestamped file in the current directory and the path is
     shown.

3. Safety: DryRun by default.
   - The default configuration sets DryRun = $true. Public functions that
     mutate target state are expected to honour this flag and only log the
     intended action.

4. Scope split across files but not over-engineered.
   - One file per topic under src/SqlServerCpy/Public/. No private helper
     sprawl, no nested subfolders, no DI abstractions. A topic may be a
     single function or a small family.

5. Configuration defaults.
   - SourceServer = "chbbbid2"
   - TargetServer = "localhost"
   - Both overridable via the TUI or a user-local config file.

6. SSAS is a documented planned extension only.
   - Not implemented in this scaffold. See docs/SSAS_PLANNED.md.

6b. SSRS copy uses the ReportService2010 SOAP API (+ REST where available)
    rather than copying the ReportServer catalog database.
    - Rationale: the ReportServer database carries encrypted columns
      (stored data-source credentials, subscription delivery credentials)
      that are bound to the source machine's symmetric key. A
      database-level copy would either drop those columns or require
      moving the SSRS encryption key out of band. The SOAP path gives
      us a version-neutral, machine-portable migration at the cost of
      having to re-enter secrets on the target.
    - Scope: the orchestrator attempts to copy EVERY class of SSRS asset
      the user can see in the portal - folders, shared data sources,
      shared datasets, reports (including linked reports), resources,
      item-level policies, role definitions (system + catalog), shared
      schedules, subscriptions, and KPIs / mobile reports via REST.
      This is a direct response to the "do not intentionally skip any
      kind of SSRS asset" requirement.
    - Failures are NOT fatal across phases: a missing principal that
      blocks SetPolicies on one item still lets the rest of the copy
      complete. Each failure is logged with an actionable hint.
    - Default endpoints:
        SourceSsrsUri = "http://chbbbid2/ReportServer"
        TargetSsrsUri = "http://localhost/ReportServer"
      Overridable in config/local.psd1 and via the TUI flow.
    - Per-asset toggles (CopySsrsFolders, CopySsrsReports, ...) default
      to $true; flip individual flags to $false only when an asset type
      is known to be unsupported in the target environment.

6c. Login copy: skip-prefix filter.
    - config/default.psd1 declares LoginSkipPrefixes =
        ('NT AUTHORITY', 'NT SERVICE', 'BUILTIN', 'ADIS').
    - Match is case-insensitive and strips a single leading "<domain>\"
      qualifier, so 'MYDOMAIN\BUILTIN\Administrators' matches 'BUILTIN'.
    - Rationale: NT AUTHORITY / NT SERVICE / BUILTIN are machine-local
      principals reissued by the target OS; copying them tends to
      produce an unresolvable login. ADIS is a site-local service
      account prefix the user does not want migrated.
    - Every skipped login is logged individually (SKIP line) so the
      effect is auditable.

7. Connection security is centralized, not per-call, and applied via
   Connect-DbaInstance connection objects (not per-cmdlet flags).
   - Three config keys drive every dbatools call:
       EncryptConnection        (default $true)
       TrustServerCertificate   (default $true)
       ConnectionTimeoutSeconds (default 15)
   - Get-SqlCpyDbaConnection opens a Connect-DbaInstance handle with
     those settings baked in. Test-SqlCpyPreflight calls it for both
     source and target and caches the handles on $Config as
     _SourceConnection / _TargetConnection.
   - Public migration functions obtain the handles via
     Get-SqlCpyCachedConnection and pass them through
     Get-SqlCpyInstanceSplat (for cmdlets that take -SqlInstance) or
     Get-SqlCpyCopySplat -SourceConnection / -DestinationConnection
     (for Copy-Dba* cmdlets). This is the only pattern that reliably
     applies TrustServerCertificate to dbatools 2.x cmdlets that do
     NOT expose -TrustServerCertificate directly (Get-DbaSpConfigure,
     Copy-DbaSpConfigure, Copy-DbaLogin, Copy-DbaAgentJob, ...).
   - The raw-name helpers (Get-SqlCpyConnectionSplat without -Connection,
     Get-SqlCpyCopySplat with string -Source / -Destination) still exist
     for Connect-DbaInstance itself and for any future caller that does
     not need connection-object reuse. They retain dynamic filtering
     against Get-Command so unsupported parameter names are dropped.
   - Get-SqlCpyDbaInstance is kept as a back-compat wrapper around
     Get-SqlCpyDbaConnection.
   - TrustServerCertificate default is $true because this tool is for
     local admin-bootstrap migrations (Windows auth, LAN, self-signed
     server cert). That disables CA/hostname validation and is therefore
     MITM-unsafe over untrusted networks. README calls this out; users
     targeting production are expected to flip it off in config/local.psd1
     and install a properly chained cert on the server.

8. Preflight before migration steps.
   - Test-SqlCpyPreflight runs Connect-DbaInstance against both source
     and target under the configured TLS settings and reports actionable
     hints for: missing dbatools, untrusted certificate chain, auth
     failure, network / host unreachable.
   - On success it stores the opened connection objects on
     $Config._SourceConnection / $Config._TargetConnection so that
     Compare / Copy steps reuse the *same* authenticated handle and
     cannot diverge in trust behaviour.
   - The TUI invokes preflight before every migration action (options
     1-6). This replaces the silent WARN + empty result we previously
     saw from Get-DbaSpConfigure when the certificate was not trusted;
     the user now gets either a clean "Preflight OK" line or a concrete
     error with a next-step hint.

Observed issue resolved
-----------------------

- WARNING: [Get-DbaSpConfigure] Failure | The certificate chain was
  issued by an authority that is not trusted   (STEP 1, chbbbid2)
  Root cause: dbatools -> Microsoft.Data.SqlClient 4.x defaults to
  Encrypt=Mandatory with strict chain validation. Get-DbaSpConfigure
  in dbatools 2.x does NOT expose -TrustServerCertificate. The splat
  helper's dynamic filter (added as the fix for the earlier
  ConnectionTimeout binding error below) therefore correctly dropped
  -TrustServerCertificate from the Get-DbaSpConfigure call, which also
  silently dropped trust and reinstated strict chain validation on a
  server (chbbbid2) with a self-signed / internal-CA cert.
  Fix shipped here: apply TrustServerCertificate through a pre-built
  Connect-DbaInstance connection object and pass THAT object as
  -SqlInstance / -Source / -Destination to downstream cmdlets.
  dbatools reuses the open connection - trust decision and all - even
  though the cmdlet itself has no -TrustServerCertificate parameter.
  Implementation:
    * Get-SqlCpyDbaConnection opens a Connect-DbaInstance handle with
      Encrypt/Trust/Timeout applied at the connection level.
    * Test-SqlCpyPreflight opens both source and target handles this
      way and caches them on the $Config hashtable under keys
      _SourceConnection / _TargetConnection.
    * Get-SqlCpyCachedConnection returns the cached handle (or opens
      a fresh one on a cache miss) for any caller that wants to run a
      downstream dbatools cmdlet.
    * Get-SqlCpyInstanceSplat wraps the handle into a splat for
      cmdlets that take -SqlInstance (Get-DbaSpConfigure, Get-DbaLogin,
      Get-DbaAgentJob, Invoke-DbaQuery).
    * Get-SqlCpyCopySplat now accepts -SourceConnection /
      -DestinationConnection; when supplied, raw name strings are
      replaced by the connection objects and trust/encrypt flags are
      omitted from the emitted splat.
  Consequence: a successful Preflight is now load-bearing - it
  guarantees that every subsequent TUI step (Compare server
  configuration, Copy logins, Copy agent jobs, etc.) uses the exact
  same TLS/trust decision Preflight validated.

- ERROR: Source connection failed: A parameter cannot be found that
  matches parameter name 'ConnectionTimeout'.
  Root cause: the initial scaffold unconditionally added
  -ConnectionTimeout to every splat. dbatools 2.x does not expose that
  parameter on several cmdlets we call (Get-DbaSpConfigure,
  Copy-DbaSpConfigure, Copy-DbaLogin, Copy-DbaAgentJob,
  Copy-DbaSsisCatalog). Splatting the missing parameter aborted TUI
  action 1 ("Compare server configuration") at preflight.
  Fix shipped here: Get-SqlCpyConnectionSplat and Get-SqlCpyCopySplat
  accept -CommandName and filter the returned hashtable against the
  named command's real parameter set (via Get-Command). If the command
  accepts -ConnectionTimeout, ConnectTimeout, or StatementTimeout the
  configured value is routed to whichever name is supported; if none
  is, the timeout is silently skipped for that call. That filtering
  is retained for raw-name calls but is no longer the primary defence
  against the cert-trust failure above - the connection-object path is.

Observations
------------

- dbatools covers Agent jobs, logins, server configuration compare, and
  schema scripting cleanly. The scaffold wires these as the primary path.
- SSIS catalog (SSISDB) object transfer is partially covered by dbatools
  (Copy-DbaSsisCatalog exists but has historically had edge cases); the
  SqlServer module's Integration Services provider + MOM-style API is a
  more reliable fallback for SSIS environments and references. Expect to
  need both.
- "Schema-only" database copy is handled by scripting objects from the
  source (dbatools: Export-DbaScript / Export-DbaDbTableData -SchemaOnly,
  or SMO Scripter) and applying them to the target. Beware of dependency
  ordering (schemas before tables, tables before foreign keys, etc.) -
  prefer the built-in ordering in dbatools' Export-DbaScript where
  possible.
- Server configuration compare/equalize should use Copy-DbaSpConfigure /
  Export-DbaSpConfigure as the baseline and supplement with explicit
  checks for items that sp_configure does not expose (e.g. tempdb file
  layout, trace flags, collation).

9. Schema-only database copy: full object model, no security, no data.
   - Requirement: produce a target database that has the same shape as
     the source but no row data and no security principals. That is the
     point of the "Copy selected databases (schema-only, no data)" TUI
     action.
   - Coverage: the copy scripts schemas, tables (columns, PK/UK/CK,
     defaults, computed columns), foreign keys, indexes (incl. filtered,
     columnstore, XML), views, scalar/inline-TVF/multi-statement-TVF/CLR
     functions, T-SQL + CLR stored procedures, DML triggers (inline with
     tables) and DDL triggers (db-scope), sequences, synonyms, alias /
     CLR / table-valued user-defined types, XML schema collections,
     partition functions and schemes, full-text catalogs and indexes,
     legacy default/rule objects, CLR assemblies, and Service Broker
     queue definitions. The combined script begins with a
     "CREATE DATABASE IF NOT EXISTS" guard so it can be replayed against
     an empty target.
   - Security categories EXCLUDED, per user instruction: users, database
     roles, application roles, permissions, role memberships, database
     audit specs, audits, credentials, asymmetric / symmetric keys,
     certificates, master key, cryptographic providers, security
     policies. The list lives in
     Get-SqlCpySchemaOnlySecurityExcludedTypes and the scripting options
     set Permissions = $false / IncludeDatabaseRoleMemberships = $false
     / LoginSid = $false to force this at the SMO layer.
   - Data categories EXCLUDED, per user instruction: the scripting
     options set ScriptData = $false, and the copy never calls
     Copy-DbaDbTableData, Write-DbaDataTable, or BCP. A syntax test
     asserts the module contains no data-movement commands.
   - Ordering: phases are emitted in a dependency-safe order (schemas
     first, tables with DRI and indexes inline, FKs/FullText/DDL
     triggers later). SMO WithDependencies is also on, so Scripter
     pulls referenced objects inside a phase.
   - Artifacts: per-phase .sql files are written under
     <output>/<db>/NN_Phase.sql plus a combined <output>/<db>.sql. In
     DryRun the artifacts are produced but not applied; the log calls
     out both paths.
   - Caveats specific to this action:
       * Encrypted modules (WITH ENCRYPTION) cannot be scripted via
         SMO .Script() - the body is unreadable. The scaffold logs a
         WARN per affected object and continues. If the target needs
         encrypted modules, decrypt on the source first or rebuild
         from source control.
       * Filegroups, physical .mdf/.ldf paths, and data-file layout
         are NOT replayed - the combined script issues a simple
         CREATE DATABASE with server defaults. Partition schemes emit
         their filegroup names; those filegroups must exist on the
         target or CREATE PARTITION SCHEME will fail. Create target
         filegroups manually if the source is partitioned.
       * Full-text catalogs/indexes require the Full-Text Search
         feature on the target. The copy will script them regardless;
         the apply step fails cleanly if the feature is missing.
       * Service Broker queues script their definitions only.
         Contracts, services, routes, and remote service bindings are
         not in scope for this action.
       * Cross-database references (three-part names, linked-server
         references) are scripted verbatim - the referenced database
         or linked server must exist on the target or the object
         becomes invalid (SQL Server still creates it; runtime use
         will fail).
       * Permissions / role memberships / explicit GRANTs tied to
         application roles are lost by design. Re-run the login copy
         action first (option 3) and re-apply ownership/permission
         post-steps manually if you need them.
       * SMO property names drift slightly between SQL versions; the
         options builder only sets properties that exist on the
         returned object, so missing properties are tolerated. If a
         category's collection is unavailable the phase is logged as
         skipped.
       * Schema phase has a dedicated fallback. We observed on SQL
         2022 / dbatools 2.1.24 (dwcontrol) that $schema.Script($opts)
         throws
             Script failed for Schema 'A00'.
         for every schema, even those whose sys.schemas owner is plain
         dbo and whose $schema.Script() / $schema.Script($opts) work
         interactively. Get-SqlCpySchemaScriptLines tries
         .Script($opts), then .Script(), then emits a manual
             IF SCHEMA_ID(N'<name>') IS NULL
                 EXEC(N'CREATE SCHEMA [<name>] AUTHORIZATION [dbo]');
         statement via Format-SqlCpySchemaCreateStatement. The
         AUTHORIZATION [dbo] choice is intentional and aligned with
         the "ignore security" requirement: replaying the source owner
         would drag in principals that the schema-only copy refuses to
         script. Names are escaped for both string-literal ('' doubling)
         and bracket-identifier (]] doubling) forms, so schemas like
         O'Brien, weird]name, or $money round-trip correctly. When the
         manual fallback is used the module logs a WARN line with the
         inner SMO exception so the underlying SMO failure is still
         visible.
       * DryRun is the default and the artifacts are written anyway,
         so the user can review exactly what would be applied before
         flipping DryRun off.
       * Tables phase has three modes. The default is a FAST IN-PROCESS
         per-table walk. A previous release defaulted to an isolated
         child-runspace mode with a per-table timeout; that mode turned
         out to add ~80-250 seconds of overhead per table on real
         sources (runspace creation + SMO re-probe + cross-runspace
         serialization), making the Tables phase unusably slow. The
         isolated mode is retained as opt-in for sources with a known
         hanging table:

           - SchemaOnlyTableScriptMode = 'InProcess' (default) iterates
             tables one at a time in the main runspace, reusing the
             already-connected SMO Database and Table objects, and logs:
                 [tables] phase starting: count=N mode=InProcess excluded_config=K
                 [tables] fast in-process mode: no hard per-table timeout
                 [table]  scripting [schema].[table] object_id=OBJ
                 [table]  done      [schema].[table] in S.ss (L lines)
                 [tables] phase done in T.tt
                 [combine] 13_Tables: appended N chars in T.tt
             before and after each table, plus parent-side phase-start /
             phase-done / combine-timing lines so the unexplained gaps
             that previously appeared between tables are now visible.
           - 'FastPerTable' and legacy 'PerTable' are aliases for
             'InProcess' (the previous meaning of 'PerTable' was the
             isolated timeout mode; it was renamed rather than removed
             so existing configs keep working).
           - SchemaOnlyTableScriptMode = 'Isolated' is the opt-in
             child-runspace path. Runs each .Script($opts) in a child
             PowerShell runspace; when SchemaOnlyTableScriptTimeoutSeconds
             elapses the runspace is stopped, the table is added to
             <output>/<db>/_skipped_tables.txt as TIMEOUT, and the
             phase continues. Best-effort: if SMO is blocked inside a
             native SqlClient socket read the TCP call may linger in the
             background, but the main runspace is freed. This mode is
             dramatically slower than 'InProcess' because every table
             incurs the runspace + SMO overhead, so only enable it when
             the source actually has a hanging table.
           - SchemaOnlyTableScriptTimeoutSeconds (default 300) ONLY
             applies in 'Isolated' mode. Setting it in 'InProcess' mode
             is a no-op; a hung table in 'InProcess' mode will hang the
             phase (the prior isolated-default sacrificed healthy-
             source throughput to defend against this and it was the
             wrong trade-off).
           - SchemaOnlyExcludeTables (default @()) is an explicit skip
             list that works in every mode. Entries accept any of:
             '[schema].[table]', 'schema.table', bare 'table' (matches
             any schema), or a numeric sys.objects.object_id. Matching
             is case-insensitive and strips brackets. Known-hanging
             tables on a given source should be listed here rather than
             relying on the Isolated-mode timeout to catch them.
           - SchemaOnlyTableScriptMode = 'Collection' is the legacy
             whole-collection path ($db.Tables scripted in one call).
             Fast in aggregate but no progress output and a single
             pathological table blocks the whole phase.
           - Every run writes <output>/<db>/_skipped_tables.txt - even
             when nothing was skipped - so operators always have a
             one-glance record of what the Tables phase did.
           - object_id logging. SMO Table.ObjectId was observed to
             return 0 in log lines (possibly a dbatools wrapper or
             cross-runspace serialization artefact on the former
             isolated path). The Tables phase now queries sys.tables
             once at phase start to build a '[schema].[table]' ->
             object_id map and uses it as a fallback whenever
             $table.ID / $table.ObjectId come back as 0. When neither
             is available the log line prints object_id=- rather than
             the misleading 0.

Caveats
-------

- This is a scaffold. Functions contain TODO markers where a live SQL
  Server is required to validate the precise cmdlet invocation, parameter
  set, and error handling. Do NOT treat the current code as production
  ready for actual migrations.

- Credentials. The scaffold passes -SqlCredential through to dbatools
  when provided, but does not manage secret storage. Use Windows auth or
  a secret manager upstream.

- Idempotency. Not all dbatools copy cmdlets are safely re-runnable.
  Always run with DryRun first and review planned changes.

- Version drift. dbatools and SqlServer module surfaces change between
  versions. Pin minimum versions in DEPENDENCIES.md and consider
  Import-Module -RequiredVersion in production.

- Clipboard on non-Windows hosts / remote sessions: Set-Clipboard may
  not be available. The error screen falls back to writing a timestamped
  error file in the current directory.

- Logging is to the screen only by default. If persistent logs are
  desired, add a Start-Transcript at the top of Start-SqlServerCopy.ps1.
